Project Layout

Project Layout is a small utility library for describing and working with conventional project directory layouts (sources, resources, tests, docs, etc.) in a consistent way. It is intended for build tooling and plugins that need to locate well-known folders reliably across different projects.

What the library provides

The library models a repository as a project root directory combined with a set of conventional, root-relative folders such as production sources, test sources, resources, documentation roots, and optional nested modules. Layout implementations can infer these folders by reading build metadata (for example Maven pom.xml, Gradle settings/build files, JavaScript workspace configuration, or Python pyproject.toml) and may use filesystem scanning as a fallback.

Consumers typically detect the appropriate layout for a given directory and then resolve the returned root-relative paths against that directory when performing filesystem operations.

Packages

org.machanism.machai.project

Facilities for discovering, describing, and processing a source-code project rooted at a filesystem directory. It provides orchestration for detecting a layout and scanning a repository (including nested modules) while delegating per-project behavior to a processor.

  • Detect a project’s on-disk convention and expose it as a ProjectLayout via ProjectLayoutManager.detectProjectLayout(File).
  • Traverse a project root and invoke a per-project hook via ProjectProcessor.

org.machanism.machai.project.layout

APIs to describe a repository's on-disk layout. The central type is ProjectLayout, which exposes conventional directories for sources, tests, resources, documentation, and optional module directories.

Concrete implementations provide build-ecosystem-specific conventions, including:

  • MavenProjectLayout (reads pom.xml via PomReader)
  • GragleProjectLayout (uses the Gradle Tooling API)
  • JScriptProjectLayout (parses package.json workspaces)
  • PythonProjectLayout (reads pyproject.toml metadata)
  • DefaultProjectLayout (filesystem-based fallback)

Typical workflow

  1. Detect a ProjectLayout for a project root directory.
  2. Query the layout for conventional directories (returned as root-relative paths).
  3. Resolve root-relative paths against the project root before filesystem access.
  4. Optionally traverse a repository and process each discovered project/module using a ProjectProcessor.

Examples

Detect a layout and access conventional folders

java.io.File projectDir = new java.io.File("C:\\path\\to\\project");

org.machanism.machai.project.layout.ProjectLayout layout =
    org.machanism.machai.project.ProjectLayoutManager.detectProjectLayout(projectDir);

java.util.List<String> sources = layout.getSources();
java.util.List<String> tests = layout.getTests();
java.util.List<String> documents = layout.getDocuments();
java.util.List<String> modules = layout.getModules();

// Resolve root-relative paths against the project directory before filesystem access.
for (String rel : sources) {
  java.io.File dir = new java.io.File(projectDir, rel);
  // ...
}

Scan a repository tree with a processor

java.io.File repoDir = new java.io.File("C:\\path\\to\\repo");

org.machanism.machai.project.ProjectProcessor processor = /* your implementation */ null;
processor.scanFolder(repoDir);

// The per-project hook is invoked for the root and for each discovered module.

Notes

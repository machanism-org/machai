Bindex Core

Bindex Core provides core functionality for bindex metadata management, including generation, registration, library selection, and project assembly. It enables automated handling of library metadata to support efficient discovery, integration, and assembly workflows within the Machanism ecosystem.

The module focuses on storing and querying org.machanism.machai.schema.Bindex documents that describe reusable projects and libraries. It combines MongoDB-based persistence with GenAI-assisted classification and embedding workflows so callers can register Bindex records, search them semantically, and expose these capabilities through tool-based integrations.

What a Bindex represents

A Bindex is a structured JSON document, commonly stored as bindex.json, that captures the identity, descriptive metadata, classification facets, and dependency relationships of a project or library. In this module, Bindex data is used as the unit of registration and retrieval.

  • Identity: logical identifiers, names, and versions used to reference a library.
  • Classification: domains, layers, languages, and integrations used for semantic matching.
  • Dependencies: relationships to other Bindex entries that allow transitive expansion.

Package structure

The API is centered on the org.machanism.machai.bindex.core package and its AI-focused subpackage.

  • org.machanism.machai.bindex.core provides the core repository and selection services. It contains MongoDB-backed persistence, Bindex registration, semantic retrieval, dependency traversal, and helper logic for converting classification data into embeddings and search queries.
  • org.machanism.machai.bindex.ai.tools provides function-tool adapters that expose Bindex retrieval, recommendation, and registration capabilities to supported GenAI providers.

Core responsibilities

  • Persistence through org.machanism.machai.bindex.core.MongoBindexRepository, which stores serialized Bindex payloads in MongoDB and retrieves them by logical identifiers.
  • Semantic library selection through org.machanism.machai.bindex.core.Picker, which classifies natural-language requests, generates embeddings, performs vector search, and returns matching Bindex entries.
  • AI tool integration through org.machanism.machai.bindex.ai.tools.BindexFunctionTools, which registers provider tools for fetching Bindex metadata, recommending libraries, and registering Bindex files.

Typical workflow

  1. Create or update a bindex.json document that describes a reusable project or library.
  2. Register the document in the configured MongoDB-backed repository.
  3. Submit a natural-language request describing required capabilities.
  4. Classify the request, generate embeddings, and perform vector search against registered Bindex metadata.
  5. Load matching Bindex records and, when needed, traverse dependency references for transitive context.

Configuration and runtime behavior

  • Java source compatibility targets release 8.
  • MongoDB access is configured through the project configurator, including optional BINDEX_REPO_URL and registration credentials.
  • GenAI and embedding providers are resolved from configuration and used to classify requests and generate vector embeddings.
  • The module depends on generated schema classes in org.machanism.machai.schema for Bindex and classification data structures.

Main types

  • org.machanism.machai.bindex.core.MongoBindexRepository manages low-level MongoDB persistence for serialized Bindex documents.
  • org.machanism.machai.bindex.core.Picker orchestrates registration, classification, semantic search, score handling, and dependency collection.
  • org.machanism.machai.bindex.ai.tools.BindexFunctionTools exposes Bindex operations as callable tools for AI-assisted workflows.

Usage examples

Register a Bindex and later retrieve recommendations for a free-text requirement.

Intended usage

This module is intended for applications and automation workflows that need a searchable registry of project and library metadata. It is especially useful when software components must be selected from descriptive requirements rather than exact coordinates, or when AI-driven tooling needs structured access to reusable project knowledge.

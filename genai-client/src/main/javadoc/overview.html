GenAI Client

GenAI Client is a Java library designed for seamless integration with Generative AI providers. It offers a provider-agnostic API for building and executing LLM requests (instructions, prompts, optional inputs, embeddings, and function tools), while keeping applications isolated from vendor SDKs and transport details.

What this module provides

  • A small public contract for issuing GenAI requests and obtaining responses and token usage.
  • Provider resolution from Provider:Model identifiers (for example OpenAI:gpt-4o-mini), including a default-provider fallback when the prefix is omitted.
  • Built-in provider integrations for OpenAI, Google Gemini, EPAM CodeMie, Anthropic Claude, and a no-op provider for offline/disabled environments.
  • An optional, curated tool system that can expose controlled host capabilities (file access, HTTP/REST calls, command execution) to model tool calls.

Package overview

The library is organized around a provider-agnostic surface, a manager layer for provider resolution and usage aggregation, concrete provider implementations, and optional host-integrated tools.

  • org.machanism.machai.ai – root package that defines the public abstractions used to build and execute LLM/GenAI requests while isolating applications from vendor-specific SDKs.

    • Groups the manager API/SPI, provider implementations, and the tool system under a consistent surface.

  • org.machanism.machai.ai.manager – manager API and SPI for selecting, configuring, and operating provider integrations.

    • Defines the provider contract (Genai), provider resolution and usage aggregation (GenaiProviderManager), a delegating adapter base class (GenaiAdapter), and the per-invocation usage value object (Usage).
    • Provider selection commonly uses Provider:Model; when the prefix is omitted, the manager falls back to the default provider.

  • org.machanism.machai.ai.provider.openai – OpenAI-backed implementation.

    • Translates MachAI prompts, tools, and usage reporting to OpenAI request/response types.
    • Provides embeddings via the common provider contract.

  • org.machanism.machai.ai.provider.gemini – Google Gemini provider integration.

    • Adapts the provider contract to Gemini models.
    • Collects instructions/prompts and (where supported) registers tools and dispatches tool calls.

  • org.machanism.machai.ai.provider.codemie – EPAM CodeMie integration.

    • Authenticates using an OIDC token endpoint, then configures an OpenAI-compatible backend.
    • Delegates to OpenAI/Gemini/Claude based on the configured model prefix.

  • org.machanism.machai.ai.provider.none – no-op provider for environments where external model calls are disabled.

    • Accumulates prompts and can persist inputs/instructions to log files for auditing.
    • Unsupported features (for example embeddings) throw UnsupportedOperationException.

  • org.machanism.machai.ai.tools – host-integrated function tools.

    • Discovers tool installers (typically via ServiceLoader) and applies them to provider instances.
    • Provides controlled wrappers for common host capabilities such as file access, HTTP/REST calls, and command execution.

Typical workflow

The example below shows the common flow: resolve a provider, optionally expose tools, set instructions and prompts, execute, then read usage. 

{@code
Configurator conf = ...;

// Resolve Provider:Model (for example: "OpenAI:gpt-4o-mini").
Genai provider = GenaiProviderManager.getProvider("OpenAI:gpt-4o-mini", conf);

// Optional: expose host-side tools (file/web/command helpers).
FunctionToolsLoader tools = FunctionToolsLoader.getInstance();
tools.setConfiguration(conf);
tools.applyTools(provider);

provider.instructions("You are a helpful assistant.");
provider.prompt("Summarize this repository.");

String answer = provider.perform();
Usage usage = provider.usage();
}

Operational notes

  • Provider instances maintain mutable, per-request state. Prefer one provider instance per request (or synchronize externally).
  • Tool execution happens in the host process. Apply tools selectively and configure security checks (for example allowed commands and allowed paths) according to your deployment environment.
  • Usage aggregation is available via GenaiProviderManager; applications can add each invocation's Usage and log aggregated totals.

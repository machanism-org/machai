GenAI Client

GenAI Client is a Java library designed for seamless integration with Generative AI providers. It offers foundational prompt management and embedding capabilities, enabling AI-powered features across Machai modules. The library simplifies interactions with AI services, supporting advanced use cases such as semantic search, automated content generation, and intelligent project assembly within the Machanism ecosystem.

Purpose and scope

This module provides a provider-agnostic API for working with generative AI services from Java applications. It centralizes provider selection, request composition, optional host-side tool registration, embedding generation, and usage reporting so that callers can integrate multiple model backends through a consistent contract instead of coding directly against vendor-specific SDKs.

The project is structured around a small public abstraction surface and a set of provider implementations. Higher-level code typically resolves a provider from a Provider:Model identifier, configures instructions and prompts, optionally exposes local tools such as file system or HTTP integrations, performs the request, and records token usage.

Architecture overview

The root package org.machanism.machai.ai defines the overall API surface and describes the interaction flow used across the module. Provider resolution and cross-request token accounting are handled by org.machanism.machai.ai.manager, whose central entry point is org.machanism.machai.ai.manager.GenaiProviderManager. The provider abstraction itself is defined by org.machanism.machai.ai.provider.Genai, with GenaiAdapter offering a delegating base for adapter or decorator-style implementations.

Concrete integrations live under org.machanism.machai.ai.provider. These include an OpenAI implementation, an EPAM CodeMie integration that authenticates and routes to downstream providers, skeletal Gemini and Claude implementations, and a no-operation provider for offline or disabled execution paths. Tool registration infrastructure is provided by org.machanism.machai.ai.tools, where service-loaded contributors can register executable functions with compatible providers.

Package structure

  • org.machanism.machai.ai – root package for the provider-neutral client API.
    • Describes the overall purpose of the module and the typical flow for resolving providers, building requests, and handling responses.
  • org.machanism.machai.ai.manager – provider resolution and usage tracking.
    • Resolves provider implementations from identifiers such as OpenAI:gpt-4o-mini or fully qualified class names.
    • Initializes providers with configuration, preserves selected chat-model information, and aggregates Usage records for logging and cost analysis.
  • org.machanism.machai.ai.provider – common provider contracts and shared abstractions.
    • Defines the Genai lifecycle, including initialization, prompting, execution, embedding generation, tool registration, and cleanup.
    • Contains the shared adapter type used to forward operations to another provider implementation.
  • org.machanism.machai.ai.provider.openai – OpenAI Responses API integration.
    • Builds requests from instructions, prompts, files, and registered tools; handles iterative tool calls; supports embeddings; and maps token counts into MachAI usage metrics.
  • org.machanism.machai.ai.provider.codemie – CodeMie authentication and provider routing.
    • Obtains OAuth 2.0 access tokens from the CodeMie OIDC endpoint, configures an OpenAI-compatible base URL, and routes requests to OpenAI-, Gemini-, or Claude-oriented providers based on model prefixes.
  • org.machanism.machai.ai.provider.gemini – Gemini provider package.
    • Defines the Gemini-backed implementation surface for adapting the common provider workflow to Google Gemini models.
  • org.machanism.machai.ai.provider.claude – Claude provider package.
    • Defines the Anthropic Claude integration point used to align Claude models with the shared provider contract.
  • org.machanism.machai.ai.provider.none – offline and no-operation execution.
    • Collects prompts without invoking a model, can persist instructions and inputs to local files, returns no generated response, and reports zero usage.
  • org.machanism.machai.ai.tools – host-side function tool registration.
    • Defines tool contributor contracts, a service-loader-based installer, and executable tool functions that expose controlled local capabilities to compatible providers.

Configuration and runtime behavior

Provider implementations are initialized from application configuration and use that configuration to resolve model names, credentials, endpoint locations, timeout settings, token limits, and optional logging behavior. OpenAI support uses provider-specific settings such as API keys and base URLs, while CodeMie support derives access tokens from user or client credentials before delegating to a downstream provider implementation.

Tool registration is intentionally separated from provider resolution. This allows applications to expose local capabilities only when required and only for providers that support tool or function calling. Because these tools may execute commands, access files, or call remote endpoints through the host process, deployments should apply careful configuration and access controls.

Typical workflow

  1. Resolve a provider using a model identifier such as OpenAI:gpt-4o-mini.
  2. Initialize the provider with application configuration and model-specific settings.
  3. Optionally apply service-loaded host tools through FunctionToolsLoader.
  4. Supply system instructions, prompts, and optional file or URL inputs.
  5. Execute the request, inspect the generated response, and record usage metrics.

{@code Configurator conf = ...; Genai provider = GenaiProviderManager.getProvider("OpenAI:gpt-4o-mini", conf); FunctionToolsLoader tools = FunctionToolsLoader.getInstance(); tools.setConfiguration(conf); tools.applyTools(provider); provider.instructions("You are a helpful assistant."); provider.prompt("Summarize this repository."); String answer = provider.perform(); GenaiProviderManager.addUsage(provider.usage()); GenaiProviderManager.logUsage(); }

Usage notes

  • Provider instances are generally mutable and should be treated as request-scoped objects rather than shared singletons.
  • The no-operation provider is useful for tests, offline execution, and prompt logging workflows where application control flow must remain intact without contacting external AI services.
  • Embedding support and advanced runtime behavior depend on the selected provider implementation.
  • Aggregated usage reporting in the manager layer helps track overall token consumption across multiple requests.

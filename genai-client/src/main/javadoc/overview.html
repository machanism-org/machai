GenAI Client

GenAI Client is a Java library designed for seamless integration with Generative AI providers. It offers foundational prompt management and embedding capabilities, enabling AI-powered features across Machai modules. The library simplifies interactions with AI services, supporting advanced use cases such as semantic search, automated content generation, and intelligent project assembly within the Machanism ecosystem.

Purpose and scope

GenAI Client provides the provider-neutral integration layer used by MachAI applications to interact with large language models and related AI services without binding application code to a single vendor SDK, authentication flow, endpoint, or execution mode. It centralizes provider resolution, model selection, prompt and instruction collection, optional host-side function tools, embedding generation, request execution, and token-usage reporting.

The library is intended for request-oriented workflows. Callers resolve a concrete provider from a model identifier, initialize it with runtime configuration, add instructions, prompts, files, URLs, or input logs, optionally register controlled local tools, execute the request, and then aggregate the returned usage metrics.

Architecture overview

The root package org.machanism.machai.ai defines the high-level client API and organizes the manager, provider, implementation, and tool packages. Provider resolution and cross-request token accounting are handled by org.machanism.machai.ai.manager, whose main entry point is org.machanism.machai.ai.manager.GenaiProviderManager. The common provider contract is org.machanism.machai.ai.provider.Genai, and GenaiAdapter supplies a delegating base for adapter and decorator implementations.

Concrete providers live below org.machanism.machai.ai.provider. The OpenAI implementation submits prompts, instructions, files, and registered tools through the OpenAI Responses API and also supports embeddings. The CodeMie provider authenticates with EPAM CodeMie and configures an OpenAI-compatible downstream endpoint. The no-operation provider preserves the provider workflow for disabled, offline, and testing scenarios without contacting an AI backend. Tool infrastructure in org.machanism.machai.ai.tools discovers service-loaded contributors and registers executable callbacks with compatible providers.

Package structure

  • org.machanism.machai.ai – root package for the provider-agnostic generative AI client API.
    • Coordinates provider management, provider contracts, concrete backend integrations, host-side tools, and usage monitoring.
    • Documents the standard flow for resolving a provider, configuring a request, executing it, and reporting token consumption.
  • org.machanism.machai.ai.manager – provider resolution and token-usage aggregation.
    • Resolves identifiers such as OpenAI:gpt-4o-mini into concrete provider classes.
    • Supports fully qualified custom provider class names and falls back to the none provider for model-only identifiers.
    • Stores the selected chatModel in configuration and aggregates immutable Usage records for summary logging.
  • org.machanism.machai.ai.provider – common provider contracts and shared abstractions.
    • Defines initialization, instructions, prompts, execution, embeddings, input logging, tool registration, usage inspection, and cleanup.
    • Provides a delegating adapter for wrapping or decorating existing provider implementations.
  • org.machanism.machai.ai.provider.openai – OpenAI provider integration.
    • Builds OpenAI Responses API requests from instructions, prompts, file references, and registered function tools.
    • Executes iterative tool calls until a final model message is produced, supports embedding generation, and maps provider token accounting to MachAI usage metrics.
  • org.machanism.machai.ai.provider.codemie – EPAM CodeMie integration.
    • Obtains OAuth 2.0 access tokens using password or client-credentials authentication.
    • Configures CodeMie Code Assistant as an OpenAI-compatible backend for delegated AI operations.
  • org.machanism.machai.ai.provider.none – disabled, offline, and test execution support.
    • Collects prompts, optionally writes instructions and inputs to local log files, returns no generated response, and reports zero usage.
    • Allows application control flow to remain intact when model execution is intentionally suppressed.
  • org.machanism.machai.ai.tools – service-provider infrastructure for function tools.
    • Defines tool contributor and executable callback contracts.
    • Uses Java ServiceLoader discovery to apply configured local capabilities to compatible generative AI providers.

Configuration and runtime behavior

Providers are initialized from application configuration. Common settings include the selected chatModel, provider credentials, base URLs, timeouts, output-token limits, tool-call limits, embedding model names, and input-log destinations. OpenAI uses settings such as OPENAI_API_KEY and OPENAI_BASE_URL. CodeMie uses credentials such as GENAI_USERNAME, GENAI_PASSWORD, and an optional AUTH_URL before delegating to its downstream compatible endpoint.

Tool registration is separate from provider creation. This separation lets applications expose local capabilities only for requests that require them and only for providers that support function or tool calling. Tool implementations may access host resources, call remote endpoints, or execute domain-specific logic, so production deployments should configure and restrict them deliberately.

Typical workflow

  1. Resolve a provider using a model identifier such as OpenAI:gpt-4o-mini, a custom provider class name, or a disabled/offline model selection.
  2. Initialize the provider with application configuration so provider-specific credentials, endpoints, limits, and model settings are available.
  3. Optionally apply service-loaded host tools through FunctionToolsLoader.
  4. Supply system instructions, prompts, and optional file, URL, or input-log data.
  5. Execute the request, consume the generated response or embeddings, and record usage metrics.

{@code Configurator conf = ...; Genai provider = GenaiProviderManager.getProvider("OpenAI:gpt-4o-mini", conf); FunctionToolsLoader tools = FunctionToolsLoader.getInstance(); tools.setConfiguration(conf); tools.applyTools(provider); provider.instructions("You are a helpful assistant."); provider.prompt("Summarize this repository."); String answer = provider.perform(); GenaiProviderManager.addUsage(provider.usage()); GenaiProviderManager.logUsage(); }

Usage notes

  • Provider instances are mutable and should generally be treated as request-scoped objects rather than shared singletons.
  • The no-operation provider is useful for tests, offline execution, prompt logging, and disabled AI workflows.
  • Embedding support, tool calling, token accounting details, and request options depend on the selected provider implementation.
  • Aggregated usage reporting in the manager layer helps track total input, cached input, and output token consumption across multiple requests.
  • Configuration values may contain placeholders such as ${...}; leave those placeholders available for runtime substitution by the application configuration layer.

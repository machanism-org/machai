GenAI Client

GenAI Client is a Java library designed for seamless integration with Generative AI providers. It offers foundational prompt management and embedding capabilities, enabling AI-powered features across Machai modules. The library simplifies interactions with AI services, supporting advanced use cases such as semantic search, automated content generation, and intelligent project assembly within the Machanism ecosystem.

What this module provides

  • A provider-neutral contract for building and executing requests against LLM/GenAI backends (instructions, prompts, optional files, and host-side function tools).
  • Provider resolution from Provider:Model identifiers (for example OpenAI:gpt-4o-mini), with support for resolving either short provider names or fully-qualified provider classes.
  • A small set of built-in providers (OpenAI, Gemini, CodeMie, Claude, and a no-op provider) plus token usage collection via a common Usage value object.
  • A curated tool system that can expose controlled host capabilities (file access, HTTP/REST calls, command execution) to model tool-calls.

Package overview

  • org.machanism.machai.ai – root namespace for MachAI's generative-AI integration layer.

    • Provides a stable, provider-agnostic surface for invoking LLM/GenAI backends.
    • Organizes built-in providers and host-side tools under well-defined subpackages.

  • org.machanism.machai.ai.manager – manager API and SPI for selecting, configuring and operating provider integrations.

    • Defines the core provider contract (GenAIProvider), an adapter base class (GenAIAdapter), provider resolution (GenAIProviderManager), and the per-invocation usage model (Usage).
    • Provider resolution commonly uses the Provider:Model form; if the provider prefix is omitted, a default provider may be used.

  • org.machanism.machai.ai.provider.openai – OpenAI-backed implementation using the OpenAI Java SDK.

    • Translates prompts, tools and usage reporting to OpenAI request/response types.
    • Supports embeddings via the provider contract.

  • org.machanism.machai.ai.provider.gemini – Google Gemini provider adapter.

    • Adapts the same provider contract to Gemini models.
    • Current implementation is a scaffold; some operations are unimplemented.

  • org.machanism.machai.ai.provider.codemie – EPAM CodeMie integration.

    • Authenticates via an OIDC token endpoint, then delegates to an OpenAI-compatible backend.
    • Routes to OpenAI/Gemini/Claude implementations based on model prefix.

  • org.machanism.machai.ai.provider.none – no-op provider for environments where external model calls are disabled.

    • Accumulates inputs and can persist prompts/instructions to log files for auditing.
    • Model-dependent features (for example embeddings) are unsupported.

  • org.machanism.machai.ai.tools – host-integrated function tools.

    • Discovers tool installers (typically via ServiceLoader) and applies them to provider instances.
    • Provides controlled wrappers for common host capabilities such as file access, HTTP/REST calls, and command execution.

Typical workflow

The example below shows the common flow: resolve a provider, optionally expose tools, set instructions/prompts, execute, then read usage. 

{@code
Configurator conf = ...;

// Resolve Provider:Model (for example: "OpenAI:gpt-4o-mini").
GenAIProvider provider = GenAIProviderManager.getProvider("OpenAI:gpt-4o-mini", conf);

// Optional: expose host-side tools (file/web/command helpers).
FunctionToolsLoader tools = FunctionToolsLoader.getInstance();
tools.setConfiguration(conf);
tools.applyTools(provider);

provider.instructions("You are a helpful assistant.");
provider.prompt("Summarize this repository.");

String answer = provider.perform();
Usage usage = provider.usage();
}

Operational notes

  • Provider instances generally maintain mutable, per-request state. Prefer one provider instance per request (or synchronize externally).
  • Tool execution happens in the host process. Apply tools selectively and configure security checks (for example allowed commands/paths) according to your deployment environment.
  • Usage aggregation is supported via GenAIProviderManager; applications can add each invocation's Usage and log aggregated totals.

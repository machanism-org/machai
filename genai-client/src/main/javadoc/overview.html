GenAI Client

GenAI Client is a Java library designed for seamless integration with Generative AI providers. It offers foundational prompt management and embedding capabilities, enabling AI-powered features across Machai modules. The library simplifies interactions with AI services, supporting advanced use cases such as semantic search, automated content generation, and intelligent project assembly within the Machanism ecosystem.

Purpose and scope

This module provides a provider-agnostic client API and a small runtime for resolving, configuring, and executing requests against different GenAI backends. Callers interact with stable abstractions and can switch providers by configuration (for example, selecting a different Provider:Model), rather than binding directly to a vendor SDK.

Architecture

The provider contract is defined by org.machanism.machai.ai.manager.Genai. Provider resolution and usage aggregation is handled by org.machanism.machai.ai.manager.GenaiProviderManager. Concrete provider integrations (OpenAI, Gemini, Claude, CodeMie, and a no-op provider) implement the contract.

Providers that support tool/function calling can be augmented with host-side function tools that expose controlled capabilities (file system access, command execution, and web/REST calls). Tool installers are typically discovered via java.util.ServiceLoader and applied using org.machanism.machai.ai.tools.FunctionToolsLoader.

Packages

The library is organized around the following packages.

  • org.machanism.machai.ai – public, provider-agnostic client API.
    • Groups the core abstractions used to configure and execute GenAI requests while keeping callers decoupled from vendor SDKs, authentication strategies, and transport details.
  • org.machanism.machai.ai.manager – provider contract and provider resolution.
    • Defines the provider contract (Genai), the resolver/aggregator (GenaiProviderManager), a delegating base implementation (GenaiAdapter), and per-invocation usage metrics (Usage).
    • Supports resolving providers via a short provider name (used in the Provider:Model format) or by fully-qualified class name.
  • org.machanism.machai.ai.provider.openai – OpenAI provider implementation.
    • Builds OpenAI API requests from prompts, optional system instructions, file inputs, and registered tools, then maps results into the common provider contract and usage metrics.
  • org.machanism.machai.ai.provider.gemini – Google Gemini integration.
    • Translates the provider-agnostic request state into Gemini API calls.
  • org.machanism.machai.ai.provider.claude – Anthropic Claude integration.
    • Adapts the common provider contract to Claude models.
  • org.machanism.machai.ai.provider.codemie – EPAM CodeMie integration.
    • Authenticates via an OpenID Connect (OIDC) token endpoint to obtain an access token, configures an OpenAI-compatible client, and delegates calls based on the configured model prefix.
  • org.machanism.machai.ai.provider.none – offline/no-op provider.
    • Never calls an external model; accumulates prompts/instructions and can optionally persist them to local log files for auditing.
    • Unsupported capabilities (for example embeddings) throw UnsupportedOperationException.
  • org.machanism.machai.ai.tools – host-integrated function tools.
    • Provides controlled host capabilities (file system, HTTP/REST, and command execution) to providers that support tool/function calling.
    • Tools are typically discovered via ServiceLoader and applied via FunctionToolsLoader.

Typical workflow

  1. Resolve a provider for a model identifier (for example OpenAI:gpt-4o-mini).
  2. Optionally apply host-side tools (file/web/command helpers) when tool calling is enabled.
  3. Supply system instructions and one or more prompts (and optionally files/URLs).
  4. Execute the request and record token usage.
{@code
Configurator conf = ...;

// Resolve Provider:Model (for example: "OpenAI:gpt-4o-mini").
Genai provider = GenaiProviderManager.getProvider("OpenAI:gpt-4o-mini", conf);

// Optional: expose host-side tools.
FunctionToolsLoader tools = FunctionToolsLoader.getInstance();
tools.setConfiguration(conf);
tools.applyTools(provider);

provider.instructions("You are a helpful assistant.");
provider.prompt("Summarize this repository.");

String answer = provider.perform();
Usage usage = provider.usage();
GenaiProviderManager.addUsage(usage);
GenaiProviderManager.logUsage();
}

Operational notes

  • Provider instances are mutable and typically hold per-request state (instructions, prompts/messages, tool registrations). Prefer one provider instance per request.
  • Function tools execute inside the host process. Apply them selectively and configure allow/deny checks (for example, allowed commands and allowed paths) according to your deployment environment.
  • Token usage can be aggregated across invocations via GenaiProviderManager to support logging and basic cost analysis.

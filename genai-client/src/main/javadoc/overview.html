GenAI Client

GenAI Client is a Java library designed for seamless integration with Generative AI providers. It offers foundational prompt management and embedding capabilities, enabling AI-powered features across Machai modules. The library simplifies interactions with AI services, supporting advanced use cases such as semantic search, automated content generation, and intelligent project assembly within the Machanism ecosystem.

Architecture at a glance

The library is structured around a provider-agnostic contract and a small set of pluggable provider implementations. Host-integrated “function tools” can optionally be installed to enable tool/function-calling capable models to read files, execute approved commands, or fetch web content.

Packages

The API is organized so application code can depend on stable abstractions while concrete provider integrations and host-side tools evolve independently.

  • org.machanism.machai.ai – provider-agnostic client API.
    • Root package that groups the public abstractions and runtime components needed to configure and execute LLM/ GenAI requests while keeping the rest of the application isolated from vendor SDKs and transport details.
  • org.machanism.machai.ai.manager – manager API / SPI for provider resolution and usage aggregation.
    • Defines the provider contract (Genai) and the resolver/aggregator (GenaiProviderManager), plus helper types such as GenaiAdapter and the per-invocation usage record (Usage).
    • Providers are typically selected using Provider:Model (for example, OpenAI:gpt-4o-mini). If the provider prefix is omitted, the manager falls back to the default provider.
  • org.machanism.machai.ai.provider.openai – OpenAI-backed implementation.
    • Builds OpenAI requests from prompts, optional system instructions, file inputs, and registered tools, then maps the response into the common provider contract and usage metrics.
  • org.machanism.machai.ai.provider.gemini – Google Gemini integration.
    • Adapts the common provider contract to Gemini models. Some operations may be incomplete depending on the current implementation.
  • org.machanism.machai.ai.provider.codemie – EPAM CodeMie integration.
    • Authenticates via an OIDC token endpoint and configures an OpenAI-compatible backend using the obtained access token.
    • Delegates to an underlying provider based on the configured model prefix (for example, gpt-*, gemini-*, or claude-*).
  • org.machanism.machai.ai.provider.none – offline/no-op provider.
    • Never calls an external model; accumulates prompt/instruction inputs and can optionally persist them to log files for auditing.
    • Unsupported capabilities (for example embeddings) throw UnsupportedOperationException.
  • org.machanism.machai.ai.tools – host-integrated function tools.
    • Provides controlled host capabilities (file access, HTTP/REST calls, and command execution) to providers that support tool/function calling.
    • Tool installers are typically discovered via ServiceLoader and applied to a provider instance via FunctionToolsLoader.

Typical workflow

A typical invocation resolves a provider for a model identifier, optionally applies host-side tools, supplies instructions and prompts, executes the request, then records token usage. 

Configurator conf = ...;

// Resolve Provider:Model (for example: "OpenAI:gpt-4o-mini").
Genai provider = GenaiProviderManager.getProvider("OpenAI:gpt-4o-mini", conf);

// Optional: expose host-side tools (file/web/command helpers).
FunctionToolsLoader tools = FunctionToolsLoader.getInstance();
tools.setConfiguration(conf);
tools.applyTools(provider);

provider.instructions("You are a helpful assistant.");
provider.prompt("Summarize this repository.");

String answer = provider.perform();
Usage usage = provider.usage();
GenaiProviderManager.addUsage(usage);
GenaiProviderManager.logUsage();

Operational notes

  • Provider instances maintain mutable, per-request state. Prefer one provider instance per request (or synchronize externally).
  • Function tools execute inside the host process. Apply them selectively and configure allow/deny checks (for example, allowed commands and allowed paths) according to your deployment environment.
  • Usage can be aggregated across invocations via GenaiProviderManager to support logging and basic cost analysis.

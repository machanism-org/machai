GenAI Client

GenAI Client is a Java library designed for seamless integration with Generative AI providers. It offers foundational prompt management and embedding capabilities, enabling AI-powered features across Machai modules. The library simplifies interactions with AI services, supporting advanced use cases such as semantic search, automated content generation, and intelligent project assembly within the Machanism ecosystem.

Architecture at a glance

GenAI Client exposes a provider-agnostic API for executing LLM/GenAI requests while keeping applications isolated from vendor SDKs and transport details. Providers are resolved from a model identifier, typically Provider:Model (for example, OpenAI:gpt-4o-mini). A default provider can be used when the prefix is omitted.

Some providers support tool/function calling. When enabled, host-side “function tools” can be installed to expose controlled capabilities such as file access, HTTP calls, or command execution.

Packages

The library is organized so application code depends on stable abstractions while concrete provider integrations and host-side tool installers evolve independently.

  • org.machanism.machai.ai – root client API.
    • Provider-agnostic entry point package that groups the public abstractions and runtime components used to configure and execute GenAI requests.
  • org.machanism.machai.ai.manager – provider contract, resolution, and usage aggregation.
    • Defines the provider contract (Genai), the resolver/aggregator (GenaiProviderManager), a delegating base (GenaiAdapter), and the per-invocation usage record (Usage).
    • Providers can be referenced by a short provider name (mapped to org.machanism.machai.ai.provider.<provider>.<Provider>Provider) or by fully-qualified class name.
  • org.machanism.machai.ai.provider.openai – OpenAI-backed implementation.
    • Builds OpenAI requests from prompts, optional system instructions, file inputs, and registered tools, then maps the response into the common provider contract and usage metrics.
  • org.machanism.machai.ai.provider.gemini – Google Gemini integration.
    • Adapts the common provider contract to Gemini models. Depending on the current implementation, some operations may be unimplemented.
  • org.machanism.machai.ai.provider.codemie – EPAM CodeMie integration.
    • Authenticates via an OIDC token endpoint to obtain an access token, configures OpenAI-compatible settings, and routes requests to an underlying provider based on the configured model prefix.
  • org.machanism.machai.ai.provider.none – offline/no-op provider.
    • Never calls an external model; collects prompts/instructions and can optionally persist them to local log files for auditing.
    • Unsupported capabilities (for example embeddings) throw UnsupportedOperationException.
  • org.machanism.machai.ai.tools – host-integrated function tools.
    • Provides controlled host capabilities (file system, HTTP/REST, and command execution) to providers that support tool/function calling.
    • Tool installers are typically discovered via ServiceLoader and applied to a provider instance via FunctionToolsLoader.

Typical workflow

  1. Resolve a provider for a model identifier (for example OpenAI:gpt-4o-mini).
  2. Optionally apply host-side tools (file/web/command helpers) when tool calling is enabled.
  3. Supply system instructions and one or more prompts.
  4. Execute the request and record aggregated token usage.
Configurator conf = ...;

// Resolve Provider:Model (for example: "OpenAI:gpt-4o-mini").
Genai provider = GenaiProviderManager.getProvider("OpenAI:gpt-4o-mini", conf);

// Optional: expose host-side tools.
FunctionToolsLoader tools = FunctionToolsLoader.getInstance();
tools.setConfiguration(conf);
tools.applyTools(provider);

provider.instructions("You are a helpful assistant.");
provider.prompt("Summarize this repository.");

String answer = provider.perform();
Usage usage = provider.usage();
GenaiProviderManager.addUsage(usage);
GenaiProviderManager.logUsage();

Operational notes

  • Provider instances maintain mutable, per-request state. Prefer one provider instance per request (or synchronize externally).
  • Function tools execute inside the host process. Apply them selectively and configure allow/deny checks (for example, allowed commands and allowed paths) according to your deployment environment.
  • Usage can be aggregated across invocations via GenaiProviderManager to support logging and basic cost analysis.

Ghostwriter

Ghostwriter is an advanced documentation engine that automatically scans, analyzes, and assembles project documentation using embedded guidance tags and AI-powered synthesis.

Purpose

Ghostwriter is a filesystem-first documentation and content generation tool. It traverses a project directory tree, discovers embedded @guidance: instructions in supported files, composes provider-ready prompts, and applies generated output back to the original files.

Ghostwriter supports all types of project files—including source code, documentation, project site content, and other relevant files.

High-level architecture

The processing pipeline follows a consistent sequence:

  • Select scan roots and matching rules.
  • Traverse the filesystem and choose files to process.
  • Extract guidance blocks using a file-format-aware reviewer.
  • Compose prompts (optionally using reusable Act templates).
  • Dispatch requests via {@code org.machanism.machai.ai.manager.Genai} and apply results in-place.

Package overview

org.machanism.machai.gw.processor

CLI entry point and processing pipeline orchestration. This package contains the Ghostwriter command-line entry point ({@code org.machanism.machai.gw.processor.Ghostwriter}) and the processors responsible for scanning a project directory, selecting files, and invoking a configured {@code org.machanism.machai.ai.manager.Genai} provider.

  • Directory traversal and filtering via {@code org.machanism.machai.gw.processor.AbstractFileProcessor}: excludes common build/tooling directories, supports glob/regex matching, and dispatches per-file work.
  • Provider execution via {@code org.machanism.machai.gw.processor.AIFileProcessor}: builds the prompt (including project metadata), configures provider working directory, applies function tools, and can optionally log composed request inputs.
  • Execution modes:
    • Guidance mode ({@code org.machanism.machai.gw.processor.GuidanceProcessor}): extracts embedded {@code @guidance:} directives using {@code org.machanism.machai.gw.reviewer.Reviewer} implementations and submits derived prompts.
    • Act mode ({@code org.machanism.machai.gw.processor.ActProcessor}): loads TOML-based Act templates (built-in and/or custom) and executes an Act prompt across matching files.

Processing is filesystem-based only: projects are not built and dependencies are not resolved.

org.machanism.machai.gw.reviewer

File-format-aware components that locate embedded {@code @guidance} directives in project files and convert them into normalized prompt fragments for Ghostwriter.

The main entry point is the {@code org.machanism.machai.gw.reviewer.Reviewer} SPI. Each implementation targets one or more file types, applies the relevant comment and string-literal conventions for that format, and produces a formatted fragment suitable for downstream processing.

Reviewers compute a stable project-relative path via {@code org.machanism.machai.project.layout.ProjectLayout#getRelativePath(java.io.File, java.io.File)} and format the final fragment using templates loaded from the {@code document-prompts} resource bundle.

org.machanism.machai.gw.tools

GenAI provider function tool registration for Ghostwriter-specific operations. This package supplies tool definitions (for example, Act discovery and Act inspection) that can be exposed to the configured provider during prompt execution.

Processing workflow

  1. Select scan targets (directory roots and/or matchers such as glob: and regex:).
  2. Traverse the filesystem and select files using include/exclude rules.
  3. Extract guidance with a format-specific reviewer.
  4. Compose prompts using extracted guidance and optional Act templates.
  5. Dispatch the request to the configured GenAI provider and apply the result.

Typical usage

{@code java -jar gw.jar [options]}

The {@code } argument may be a project-relative directory (for example, {@code src/main}) or a matcher expression supported by {@code java.nio.file.FileSystems#getPathMatcher(String)} (for example, {@code glob:**/*.java}).
